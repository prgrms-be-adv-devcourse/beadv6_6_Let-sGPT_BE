# 관리자 챗봇 구현 로드맵 및 진행 현황

> 기준일: 2026-07-24
>
> 이 문서는 [핵심 설계안](./DESIGN.md)과 [상세 설계안](./DETAILED_DESIGN.md)을 구현하는 순서와 현재 상태를 관리한다. 기존 Spring AI 구현의 존재 여부가 아니라 새 점진적 공개 구조의 완료 여부를 기준으로 `예정`, `진행중`, `완료`를 표시한다.
>
> 아래 1~7단계는 **구현 작업 단계**다. 상세 설계안의 요청별 0~4단계와는 구분한다.

## 1. 단계별 현황

| 단계 | 상태 | 완료 기준 |
|---|---|---|
| 1. 전체 구조와 1차 LLM 계약 확정 | 완료 | 실제 추론 서버에서 일반·도구·내부 영역·복합·보안·직전 맥락 검증 |
| 2. 상세 스키마와 2차 LLM 계약 확정 | 완료 | 여섯 영역 스키마, 부분 실패와 복합 조합을 실추론으로 검증 |
| 3. 고정 단계 오케스트레이션 구현 | 완료 | 통합 1차 요청, 병렬 도구, 상세 구조화와 사실 누적 구현 |
| 4. 내부 읽기 모델·외부 도구 통합 | 완료 | 승인된 전체 운영 질문을 새 흐름에서 조회하고 보안 경계 검증 |
| 5. 단계적 SSE와 프런트 연동 | 완료 | 먼저 완료된 자연어와 최종 답변을 중복 없이 스트리밍 |
| 6. 종단 품질·성능 최적화 | 완료 | 실제 DB·외부 API·8K 조건에서 대표 기능 품질과 시간을 확인 |
| 7. 완료 검증과 구현 리뷰 | 완료 | 백엔드·프런트 빌드, 브라우저 확인, 독립 코드 리뷰 1회와 문서 동기화 |

## 2. 1단계 — 전체 구조와 1차 LLM 계약

### 목표

전체 내부 스키마를 매번 보내거나 자연어를 규칙으로 선분기하지 않고, 첫 요청 하나로 다음을 판단할 수 있는지 확인한다.

```text
일반 질문의 직접 답변
+ 가벼운 도구 인자
+ 무거운 내부 데이터 영역 조합
```

### 완료 사항

- 일반 질문과 도구 질문을 하나의 1차 LLM 요청으로 통합
- 가벼운 실제 도구 6개와 내부 영역 선택 도구 1개 확정
- 내부 데이터 영역 6개 확정
- 직전 질문·답변 한 번만 제한적으로 전달
- 복합 질문에서 여러 도구와 데이터 영역 동시 선택
- tool call과 함께 오는 자연어의 노출 규칙 확정
- 같은 단계 병렬 실행과 사실 압축본 누적 원칙 확정
- 8K 기준 단계별 입력 분리와 고정 깊이 흐름 확정
- `Codex 5.6 sol max` 독립 읽기 전용 설계 리뷰 1회 완료

### 실추론 결과

테스트 코드를 만들지 않고 실제 로컬 추론 서버에 직접 요청했다.

| 항목 | 결과 |
|---|---:|
| 시나리오 | 19개 × 2회 = 38회 |
| 기대한 직접 답변 또는 도구·영역 선택 | 38/38 |
| 도구 포함 요청 중앙값 | 5.676초 |
| 도구 포함 요청 p95 | 8.922초 |
| 일반 답변 중앙값 | 7.653초 |
| prompt token 중앙값 / 최대 | 1,217 / 1,235 |
| 1차 프롬프트·도구 후보 크기 | 약 3,582자 |

현재 전체 관리자 프롬프트·도구 정의 11,271자보다 약 68.2% 작다. 개별 주문과 결제기한 경과 주문 도구를 첫 요청에 추가하면 중앙값이 약 0.49초 늘지만, 해당 질문에서 약 5~7초의 추가 스키마 선택 호출을 없앨 수 있어 최초 공개 대상으로 유지했다.

일반 전용 경로와 통합 요청을 각각 3회 스트리밍 비교한 결과, 통합 요청의 추가 비용은 첫 자연어 약 0.45초, 전체 약 0.32초였다. 따라서 `GeneralQuestionPolicy` 기반 이중 경로보다 통합 요청을 선택했다.

상세 계약과 표본은 [1차 요청 최적화 설계](./FIRST_REQUEST_OPTIMIZATION.md), 이전 Spring AI 방식과의 비교는 [로컬 비교 벤치마크](./SPRING_AI_LOCAL_BENCHMARK.md)에 기록한다.

### 현재 상태

1차 계약을 그대로 애플리케이션에 구현했다. 로컬 전용 추론, 2차 wire 계약, 전부 실패 답변, shard 병합과 안전한 스트리밍 노출 규칙까지 코드와 테스트에 반영했다.

## 3. 2단계 — 상세 스키마와 2차 LLM 계약

### 목표

1차에서 선택한 내부 영역만 제공해 자연어 질문을 실행 가능한 독립 조회로 구조화한다.

### 구현 사항

- 기존 분석 카탈로그를 다음 여섯 스키마 팩으로 재구성
  - `ORDER_SALES`
  - `PAYMENT_REFUND`
  - `SETTLEMENT_RECONCILIATION`
  - `MEMBERSHIP`
  - `CATALOG_INVENTORY`
  - `EVENT_SAGA_RELIABILITY`
- 공통 기간·비교·정렬·행 수 계약 분리
- 각 필드의 의미, 필수 여부, enum·형식과 제약 정의
- 상대 기간의 서버 계산과 명시 기간의 KST 시작 포함·종료 미포함 정규화
- 독립 조회별 `SUCCESS`·`FAILED` 구조화 결과 정의
- 자동 실행을 끈 `submitInternalQueryBindings` 2차 tool call 계약 검증
- 서버 발급 binding 식별자와 항목별 독립 파싱·검증
- 선택 영역 조합이 8K 예산을 넘을 때 같은 단계 병렬 분할 규칙 정의
- shard 상한을 중앙 컨텍스트 예산 설정으로 분리하고 기본값 6 적용
- primary shard 한 곳만 단계적 자연어를 만들고 shard 결과를 안정 정렬·중복 제거
- 가벼운 결과와 상세 스키마를 함께 보낸 2차 답변 품질 측정

### 실추론 검증

- 영역별 단독 질문
- 같은 영역 안의 다중 통계 질문
- 두 개 이상 영역의 복합 질문
- 모호하거나 지원하지 않는 일부 조건
- 가벼운 도구와 내부 데이터가 섞인 질문
- 모든 binding이 실패한 내부 질문의 자연어 종료
- 직전 맥락으로 기간이나 대상을 생략한 질문
- 개인정보·쓰기·자유 SQL 요청

### 완료 기준

- 지원 질문은 올바른 지표·차원·필터·기간으로 구조화
- 채울 수 없는 독립 조회만 `FAILED`가 되고 정상 조회는 보존
- 조합별 prompt token과 응답시간 측정 완료
- 최악 조합에서도 선택 영역이 조용히 누락되지 않음
- 실제 구현에 사용할 최소 스키마 계약 확정

### 구현 결과

- 여섯 도메인의 데이터셋별 지표 enum과 설명, 차원·필터·시간 필드를 닫힌 카탈로그로 분리했다.
- `metricValues`와 `metricMeanings`를 분리해 모델이 설명을 enum 값에 섞는 오류를 제거했다.
- 현재 전체 여섯 영역 조합은 약 3,583 tokens로 입력 목표 6,000 tokens 안에 들어 한 shard에서 처리된다.
- 단독·복합 질문에서 같은 조건의 여러 지표를 한 binding으로 묶고, 독립 조건은 여러 binding으로 보존한다.
- 스키마 전체 파싱 실패에만 한 번 형식 복구를 허용하며 필드 단위 실패는 다른 성공 결과를 제거하지 않는다.

## 4. 3단계 — 고정 단계 오케스트레이션

### 목표

일반 질문 1회, 가벼운 질문 2회, 내부 데이터·복합 질문 최대 3회의 고정 흐름을 애플리케이션 책임으로 구현한다.

### 구현 사항

- 일반·관리자 이중 경로를 통합 1차 요청으로 교체
- 관리자 챗봇 전 단계에 외부 폴백 없는 로컬 전용 추론 경로 적용
- 1차 응답 완료 뒤 content-only와 tool-call 모드 검증
- 초기 도구 7개의 표준 tool call 해석과 서버 검증
- 같은 단계의 가벼운 도구 병렬 실행
- 선택 영역 상세 스키마 조합과 예산 계산
- 자동 실행 없는 2차 LLM tool call 수집과 구조화 결과 검증
- `SUCCESS` Query Plan만 내부 조회로 전달
- 도구·구조화·조회 결과의 공통 사실 압축본
- 이미 사용자에게 전달한 결과 식별자 누적
- 3차 최종 답변 요청에서 도구 스키마 제거
- 빈 응답·형식 오류의 제한적 복구와 중복 실행 방지
- 내부 binding이 전부 실패해도 3차 자연어 답변 보장
- 요청 절대 deadline의 모든 자식 작업 전파와 일괄 취소

### 구조 원칙

- Spring AI는 OpenAI 호환 호출과 tool contract 연결에 사용한다.
- 단계 전환과 결과 보존은 애플리케이션이 명시적으로 관리한다.
- 프레임워크의 자동 tool loop에 전체 흐름을 숨기지 않는다.
- 계층별 책임을 분리하고 기존 포트·어댑터 의존 방향을 유지한다.
- 단계 수를 늘리는 재귀 라우팅과 도구별 LLM 재호출은 만들지 않는다.

### 완료 기준

- 네 질문 유형의 순차 왕복 깊이와 순서가 계약대로 고정
- 병렬 완료 순서와 관계없이 결과가 유실되지 않음
- 한 도구·구조화 단위 실패가 전체 성공 결과를 제거하지 않음
- 요청별 토큰 예산과 전체 deadline을 초과하지 않음
- 가까운 단위·계약 테스트 통과

### 구현 결과

- `AdminChatOrchestrator`가 일반 1회, 가벼운 2회, 내부·복합 3회의 순차 흐름을 명시적으로 소유한다.
- Spring AI 자동 tool loop 대신 라우팅, 구조화, 최종 답변 호출을 단계 전용 계약으로 분리했다.
- 초기 도구는 병렬 실행하고 결과를 공통 사실 압축본으로 바꿔 다음 단계에 전달한다.
- 2차 구조화 결과는 항목별로 검증하고 `SUCCESS` Query Plan만 병렬 조회한다.
- 큐 대기 전에 생성한 절대 deadline을 모든 단계가 공유하고 terminal 시 예약·실행 작업을 취소한다.

## 5. 4단계 — 내부 읽기 모델과 외부 도구 통합

### 재사용할 현재 자산

- 주문·결제·환불·정산·회원·상품·드롭·이벤트·사가 분석 카탈로그
- 고정 Query Plan과 전용 `ai_read` 읽기 모델
- 공개 주문번호 상태·이력·현재 사가 조회
- 결제기한 경과 `PAYMENT_PENDING` 주문 수
- 운영 문서 레지스트리
- Open-Meteo, CoinGecko와 Tavily 어댑터

### 구현 사항

- 여섯 상세 영역과 현재 데이터셋·지표·차원·필터의 매핑 검증
- “많이 팔린 상품”, “사용자가 몰린 시간”, “실패율”, “정산 예정액” 같은 운영 질문 회귀 표본 구성
- 회원·판매자 집계의 버킷별 서로 다른 주체 수와 재식별 방지 검증
- 조회 행·필드·문자 수 제한과 사실 압축 검증
- 외부 검색어·날씨 위치의 개인정보 제거와 차단 검증
- 도구 비활성·provider 실패·DB timeout의 부분 성공 확인
- `ai_read` 운영 사용 전 도메인 담당자와 데이터 의미·변경 통지·배포 순서 협의

### 보안 경계

- 특정 회원 조회 스키마 없음
- 공개 주문번호 외 내부 식별자 조회 없음
- 자유 SQL 생성·실행 없음
- 승인 view·함수 외 DB 권한 없음
- 원본 DataSource 폴백 없음
- 외부 도구로 질문 원문을 무조건 전달하지 않음
- 관리자 LLM 요청의 외부 provider 폴백 없음

### 완료 기준

지원 대상 운영 질문이 새 흐름에서 실제 데이터로 답변되고, 기존 보안·권한·부분 성공 테스트가 모두 통과한다.

### 구현 결과

- 주문·판매, 결제·환불, 정산·대사, 회원, 상품·드롭·재고, 이벤트·사가의 여섯 읽기 영역을 고정 카탈로그와 `ai_read` 조회로 연결했다.
- 회원과 판매자 보호 집계는 임의 필터 없이 단일 분류·고정 달력 기간만 허용하고, 각 기간 버킷에서 k=5를 limit 전에 적용하며 한 행만 숨겨지는 경우 보완 억제를 수행한다.
- 개별 주문은 공개 주문번호만 허용하고 존재 확인, 상태·이력·사가를 최대 100건으로 제한한다.
- 주문·상품·드롭의 공개 식별자와 상품명·가격·수량·상태는 비민감 운영 필드로 분리해 최대 20행의 제한된 개별 조회를 허용했다.
- 현재 스냅샷 데이터셋은 기간 조건 없이 조회하고, 시간 계산 파라미터는 PostgreSQL `timestamptz`로 명시했다.
- 로컬 프로필은 기존 DB 자격증명을 재사용해 읽기 모델을 자동 구성하고, 원본 테이블 기동이 늦으면
  서버를 중단하지 않은 채 재시도한다.
- 운영 read-model은 매 전체 배포의 Sync Hook에서 분리했다. DDL 산출물 또는 조회 Secret
  revision이 달라질 때만 identity가 다른 선행 Job이 실행되고, 동일 완료 Job은 재사용된다.
- 운영 AI 파드는 추론 키와 Tavily 키를 전용 Secret에서 주입받으며, 비폴백 추론 경로가 확인되기
  전에는 원격 관리자 추론을 fail-closed한다.
- Open-Meteo, CoinGecko, Tavily와 압축 운영 문서를 초기 도구로 연결했다.

## 6. 5단계 — 단계적 SSE와 프런트

### 목표

긴 내부 조회가 함께 있어도 먼저 완료된 독립 결과를 자연어로 보여주고, 이후 최종 결과를 같은 대화 흐름으로 이어서 표시한다.

### 구현 사항

- 2차의 먼저 완료된 자연어와 3차 최종 자연어 순차 전송
- 전달 완료 `segmentId`를 이용한 중복 답변 방지
- 일부 답변 뒤 실패 시 `partial=true`
- 연결 종료 시 LLM·도구·조회와 예약 작업 취소
- 기존 단순 이벤트 `started`, `status`, `delta`, `done|error` 유지
- 친절한 진행 문구, 스피너·경과 초와 부드러운 스크롤 회귀 확인
- 카드·도구명·스키마·디버그 정보 비노출

### 완료 기준

- 일반 답변은 1차 전체 응답이 content-only로 검증된 뒤 자연어 조각으로 전달
- 가벼운 결과는 2차에서 자연어로 완료
- 복합 질문은 먼저 완료된 답과 최종 답이 누락·중복 없이 이어짐
- terminal 이벤트가 정확히 하나이며 취소·timeout에서도 자원이 정리됨
- 프런트 테스트·타입 검사·프로덕션 빌드 통과

### 구현 결과

- 백엔드는 `started → status → delta+ → done|error` 계약만 유지하고 카드·도구명·스키마를 노출하지 않는다.
- 일반 답변은 1차 응답 모드를 확정한 뒤 조각으로 재생하고, 가벼운 답과 최종 답은 같은 `delta` 스트림에 순서대로 누적한다.
- 프런트는 기존 스피너·경과 시간·자동 스크롤을 유지하면서 새 capability와 직전 한 턴 계약을 반영했다.
- 프런트 전체 22개 파일, 76개 테스트와 타입 검사, 프로덕션 빌드를 통과했다.

## 7. 6단계 — 종단 품질과 성능

### 테스트 범위

- 일반·운영·외부·내부 6영역
- 개별 주문과 현재 사가
- 다중 영역과 외부+내부 복합 질문
- 직전 한 턴 참조
- 일부 구조화·도구·조회 실패
- 모든 binding 실패와 내부 조회 0건
- 개인정보·쓰기·SQL·프롬프트 우회
- 로컬 추론 장애 시 외부 provider 폴백 차단
- 8K 경계와 같은 단계 병렬 분할
- 동시 요청 1·2·3건과 실행기 포화
- SSE 취소·heartbeat·deadline·부분 종료

### 측정 항목

- 단계별 prompt token
- 1차 선택 시간
- upstream 첫 토큰과 사용자에게 검증 후 보낸 첫 자연어 시간
- 단계적 답변 시간
- 최종 완료 시간
- 도구·DB 실행 시간
- 분기·구조화·답변 정확도
- 부분 성공률과 빈 응답
- 동시 요청 처리량과 편차

기존 `2 threads, queue 8`, heartbeat 15초, 작업 deadline 170초와 emitter timeout 180초는 초기값이다. 새 흐름 측정 결과가 나온 뒤 안정성을 해치지 않는 범위에서 조정한다.

### 완료 기준

- 대표 질문 반복 성공
- 기존 구현 대비 자연어 범위와 응답시간 개선 확인
- 30초 이상 지연의 원인을 단계별로 설명 가능
- 오류와 부분 성공이 사용자에게 자연스럽게 전달
- 8K 초과와 결과 유실 없음
- 외부 폴백·중복 shard 답변·빈 terminal 답변 없음

### 실환경 결과

로컬 OpenAI 호환 추론 서버에 `reasoning_effort=none`을 적용하고, 테스트 데이터가 있는 PostgreSQL과 실제 외부 API를 연결해 측정했다.

| 대표 질문 | 완료 시간 | 결과 |
|---|---:|---|
| 엑셀이 뭐야? | 9.671초 | 일반 경로 1회 완료 |
| 부천 비 와? | 18.870초 | Open-Meteo 위치 보정·예보 완료 |
| 오늘 주문 상태별 수 | 27.166초 | 총 9건과 상태별 집계 완료 |
| 회원 역할별 수 | 28.910초 | k=5 미만 그룹 억제 안내 |
| 주문+회원 복합 | 36.133초 | 두 영역 결과 보존 |
| 개별 주문 사가 | 23.602초 | 상태·이벤트·현재 사가 완료 |
| 최근 30일 상품 판매 순위 | 35.514초 | 상품별 완료 수량 순위 |
| 최종 정산액·수수료 | 32.167초 | 731,500원·38,500원 |
| 결제 성공률·환불액 | 39.055초 | 71.43%·29,000원 |
| 정체 사가·대기 이벤트 | 34.682초 | 정체 1건·대기 0건 |

일반 답변은 약 7~12초, 가벼운 외부 도구는 약 16~24초, 내부 데이터와 복합 질문은 약 27~39초였다. DB와 외부 도구 자체보다 로컬 모델의 1차 선택·2차 구조화·최종 문장 생성이 지연의 대부분을 차지한다. 단계 상태와 SSE 자연어 조각으로 대기 중 진행을 계속 표시한다.

## 9. 구현 시작 전 고정 사항

- 자연어 질문과 자연어 답변
- 첫 요청에서 작은 스키마만 공개
- 일반 질문의 규칙 기반 선분기 없음
- 내부 데이터 상세 스키마는 선택된 영역만 공개
- 고정 의미 단계와 같은 단계 병렬 처리
- 독립 실패가 전체 성공을 제거하지 않음
- 내부 구조화가 전부 실패해도 자연어 답변으로 종료
- 회원은 집계만, 개별 주문은 공개 주문번호만
- 자유 SQL·쓰기·원본 테이블 접근 없음
- 관리자 챗봇 LLM 요청의 외부 provider 폴백 없음
- 질문·답변 저장 없음
- 카드와 내부 처리 정보 없음

2단계 상세 스키마 실추론 결과에 따라 필드와 프롬프트는 조정할 수 있지만, 위 구조와 보안 경계를 바꾸는 경우에는 구현 전에 다시 설계 결정을 확인한다.
