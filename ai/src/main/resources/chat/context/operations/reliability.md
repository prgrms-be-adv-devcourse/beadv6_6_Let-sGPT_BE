# RELIABILITY — 일일 운영·장애 점검

운영 점검은 쓰기보다 읽기 확인에서 시작한다. 챗봇 결과, 서비스 헬스, 메트릭, 로그와 저장 이벤트 중 하나만으로 원인을 확정하지 않고 서로 교차 확인한다.

일일 권장 순서:

1. Gateway와 회원·상품·주문·결제·정산·AI 서비스의 readiness/liveness 확인
2. 오늘 예정·진행·종료 드롭의 시간과 현재 파생 상태 확인
3. 주문 수와 상태 분포, 결제 대기 만료, 실패 상태 변화 확인
4. 장시간 `COMPENSATING` 사가와 재고 복원 실패 확인
5. 결제·환불 `PENDING` 정체와 PG 연동 실패 확인
6. Kafka outbox 미발행, 소비 실패·DLQ, 이벤트 처리 지연 확인
7. 재고 원장 합계가 음수 또는 최초 수량을 초과하는 징후 확인
8. 일별 대사 결과와 월 정산 gate·배치 실패 확인
9. 회원 역할·가입 경로와 상품 카테고리·생명주기 분포의 비정상 변화 확인

장애 분류 원칙:

- 주문 실패 증가: 실패 코드와 상태 전이 → 재고 차감 결과 → 결제 결과 이벤트 → 사가 순서로 확인
- 결제 대기 증가: 주문의 결제 만료 시각 → payment의 PENDING 회수 상태 → PG 조회와 이벤트 발행 확인
- 취소·환불 정체: 환불 상태 → PG 조회 확실성 → 주문 환불 이벤트 → 재고 복원 사가 확인
- 재고 이상: 드롭 파생 상태 → 총수량과 변경 이력 합계 → 주문별 중복 방어 → 보상 실패 확인
- 정산 이상: payment 조회 성공 여부 → 일별 대사 상태와 차이 유형 → 월 gate → 판매자별 배치 결과 확인

구현된 자동 감시에는 결제 PENDING·환불 PENDING 회수 스캐너, 10분 이상 보상 중 사가 경고, payment outbox 5분 이상 미발행 경고, 일별 대사와 월 정산 gate가 있다. Kafka 소비 실패는 일부 흐름에서 재시도 뒤 DLQ로 격리한다. 이 기능이 있다는 사실과 현재 정상이라는 판단은 다르다.

관리자 챗봇은 현재 조회 도구가 반환한 범위만 “지금 상태”로 말한다. 결제·정산·전체 서비스 헬스·로그·DLQ 수치가 도구 결과에 없으면 확인 절차와 담당 시스템을 안내하고 정상·장애 여부를 단정하지 않는다. 쓰기 재처리를 직접 실행하지 않는다.
