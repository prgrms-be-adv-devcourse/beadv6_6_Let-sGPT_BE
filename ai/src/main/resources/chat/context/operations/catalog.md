# OPENAT 운영 컨텍스트 라우팅 카탈로그

이 카탈로그는 항상 짧게 제공하고, 질문에 필요한 상세 문서만 추가한다. 여러 영역 질문은 관련 문서를 조합한다. 정적 문서는 현재 수치나 장애 원인을 증명하지 않으므로 실시간 사실은 반드시 조회 도구 결과를 우선한다.

| contextId | 선택 기준 | 상세 문서 |
|---|---|---|
| `PLATFORM` | 우리 플랫폼의 정체성·목적·사용자·서비스·전체 업무 흐름 | `platform.md` |
| `ORDER_PAYMENT` | 주문, 결제, 취소, 환불, 개별 주문 이력·사가, 보상 | `orders-payments.md` |
| `CATALOG_INVENTORY` | 상품, 카테고리, 드롭, 재고, 품절, 구매 한도 | `catalog-inventory.md` |
| `MEMBER_ACCESS` | 회원·판매자·관리자 역할, 가입 경로, 집계와 접근 경계 | `members.md` |
| `SETTLEMENT` | 판매 적재, 일별 대사, 월 정산, 불일치, 실패 재처리 판단 | `settlement.md` |
| `RELIABILITY` | 일일 점검, 장애 분류, 이벤트 지연, 보상 정체, 헬스·메트릭 | `reliability.md` |
| `REPORTING` | 챗봇이 조회 가능한 데이터, 기간·스냅샷 해석, 답변 한계 | `reporting.md` |
| `OFFICE_PRODUCTIVITY` | 엑셀·스프레드시트 활용, 보고서·인수인계 방식 | `office-productivity.md` |

라우팅 원칙:

- “우리 플랫폼이 뭐야?”는 `PLATFORM`만 선택한다. 재고 기술 하나만 플랫폼 전체인 것처럼 답하지 않는다.
- “관리자가 매일 뭘 봐야 해?”는 `RELIABILITY`를 기본으로 선택하고, 구체 도메인이 있으면 해당 문서를 함께 선택한다.
- 주문과 결제, 드롭과 재고처럼 원인과 결과가 연결된 질문은 두 영역을 억지로 분리하지 않고 결합 문서를 사용한다.
- 현재 건수·상태·추이는 `REPORTING`과 데이터 도구가 담당한다. 운영 문서의 절차를 현재 사실처럼 말하지 않는다.
- 선택 근거가 없으면 상세 문서를 전부 싣지 않는다. 답변에 필요한 문서는 보통 1개, 복합 질문은 2개 이내가 우선이다.
