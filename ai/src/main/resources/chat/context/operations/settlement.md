# SETTLEMENT — 판매 적재·대사·월 정산

정산 서비스는 결제 완료·환불 이벤트를 받아 정산 대상 판매 데이터를 적재하고, 판매자별 월 정산을 Spring Batch로 처리한다. 정산 대상 주문은 `READY`에서 판매자 정산 결과에 반영되면 `COMPLETED`가 된다. 월 배치 상태는 `READY`, `RUNNING`, `COMPLETED`, `FAILED`다.

일별 대사는 전날 영업일의 payment 데이터와 settlement 저장 데이터를 비교한다. 결과는 다음과 같다.

- `SUCCESS`: 해당 실행에서 비교가 통과했고 불일치가 0이라는 뜻
- `DISCREPANCY_FOUND`: 누락 또는 금액 불일치 등이 하나 이상 있다는 뜻
- `CALL_FAILED`: payment 일별 조회 자체가 실패했다는 뜻이며 정상이나 불일치 0을 의미하지 않음

필수 필드가 있는 결제·환불 항목이 settlement에 누락되면 일별 대사가 복구 적재를 시도한다. 이미 저장된 행의 금액이 다르면 덮어쓰지 않고 불일치로 기록한다. 대상 월의 모든 날짜에 일별 대사 결과가 존재하고 모두 `SUCCESS`이면서 불일치 수가 0이어야 월 정산 gate가 통과한다.

`SUCCESS`는 판매자 지급 완료나 모든 PG 거래의 최종 정합성을 증명하지 않는다. 일별 대사 결과, 월 배치 상태, 판매자별 정산 결과, payment 측 PG 대사를 각각 구분해 본다. `CALL_FAILED`를 불일치 없음으로, 배치 `COMPLETED`를 실제 송금 완료로 과장하지 않는다.

설정상 일별 대사는 02시, 월 정산은 매월 1일 03시에 실행되지만 스케줄러에 시간대가 명시되지 않았다. 배포 JVM 기본 시간대를 확인하기 전에는 한국 시간 실행이라고 단정하지 않는다.

장애 확인 순서:

1. 대상 영업일과 일별 결과 상태·불일치 수 확인
2. `CALL_FAILED`면 payment 연결과 조회 실패 확인
3. `DISCREPANCY_FOUND`면 누락, 금액 차이, 판매자 식별자 누락 유형 확인
4. 대상 월의 누락 날짜와 월 정산 gate 결과 확인
5. 월 배치와 판매자별 실패 결과 확인
6. 실제 재대사·재처리는 승인된 관리자 도구에서 수행

현재 관리자 챗봇 데이터 도구에는 결제·정산 실시간 집계가 연결돼 있지 않다. 따라서 정산 문서는 상태 의미와 점검 절차를 설명할 뿐 현재 실패 건수나 배치 성공 여부를 만들어내지 않는다.
