# 재고 게이트키퍼 설계서 (Stock Gatekeeper)

> 한정 수량 드롭(Drop)의 선착순 재고 차감을 담당하는 **상품(product) 도메인** 설계임.
> 현재는 **추상 흐름만** 정의하며, 구체적인 스펙(Redis 자료구조, Lua 스크립트 코드, DDL, TTL, API 스펙 등)은 **구현 시점에 확정**하여 반영함.
> 동시성 전략 근거: `DECISIONS.md` 결정 #7 / 멱등성 제약 근거: 결정 #8.

---

## 1. 개요 및 역할
- 주문 도메인에서 호출하는 **내부 동기 API**로, "오픈된 드롭에 차감 가능한 재고가 존재하는지"만을 판정하여 응답함.
- 사용자 세션이나 대기열은 관리하지 않으며, **주문별 재고 소진 카운트 및 이력 관리**에 집중함.

---

## 2. 핵심 전략 (결정 #7)
- **캐시(Redis) 중심의 Drop 상태 및 조회 관리:** 활성화된(OPEN 또는 오픈 임박) Drop 메타 정보와 라이브 잔여 수량을 **DROP Cache List** 형태로 캐시에 적재하여 1차로 관리함.
- **재고 변경 이력의 원장(Append-only) 관리:** 별도의 잔여 재고 스냅샷 컬럼을 두지 않고, 재고 변경 발생 시 이력을 누적하는 방식을 취함. 잔여 재고는 `total_quantity + SUM(quantity_delta)`으로 계산됨.
- **Lua 스크립트를 통한 원자성 보장:** 멱등성 검증 및 재고 확인·차감 처리를 단일 원자적 단위(Atomic Unit)로 수행하여 레이스 컨디션을 방지함.
- **RDB 기반의 진실 소스(Source of Truth) 확보:** `stock_histories` 원장 테이블을 최종 신뢰 원천으로 삼음. 캐시는 언제든 재구성 가능한 가속기로 취급하며, 정합성 의심 시 RDB 데이터를 기반으로 재워밍함.

---

## 3. 데이터 구조 (개요)
- **RDB(원장):** `drops`(메타 및 상태 관리) 테이블과 `stock_histories`(이력 누적, `DEDUCT`/`ROLLBACK`, 부호 있는 `quantity_delta` 포함, `UNIQUE(order_id, change_type)` 제약) 테이블로 구성됨.
- **캐시(Drop 객체):** 활성 Drop별 상태, 라이브 잔여 수량, 멱등 결과 등을 보유함 (세부 자료구조 및 표현 방식은 구현 시 확정).

### 4.1 요청 및 처리 흐름 (전체)
- 사용자의 선착순 구매 요청 발생 시, 주문 도메인에서 **주문을 선행 생성**하여 비즈니스 식별자를 먼저 확보함.
- 확보된 주문 및 드롭 식별자를 기반으로 상품 도메인의 내부 API를 호출하여 **재고 차감을 동기적으로 요청**함.
- 상품 도메인은 오픈 여부 및 실시간 잔여 재고를 원자적으로 검증·선점한 후, 확보 성공 또는 실패 여부를 응답하여 주문의 최종 진행 상태를 결정함.
- 재고 확인이 성공하면 주문은 결제 대기(`order_payment_pending`) 상태로 진입하며, 이후 **결제 타임아웃(Timeout)** 이 발생할 경우 시스템 안정성을 위해 상품 도메인의 재고 롤백 API를 트리거하도록 설계함.

### 4.2 캐시 워밍 (이력 집계 → Drop 객체)
- 드롭 오픈 스케줄러(Drop 오픈) 또는 콜드 스타트 시점에 `drops` 테이블의 수시 가용 상태를 조회하여, 기존 재고 이력을 집계한 뒤 **DROP Cache List**를 Redis에 적재함.
- 워밍 작업이 완료되기 전에는 해당 드롭에 대한 차감 요청을 처리하지 않음.

### 4.3 재고 감소 (API)
1. 캐시 내 활성화된 Drop 존재 여부 및 오픈 상태를 1차 검증함.
2. **Lua 스크립트 실행:** 멱등성 확인 및 재고 수량 검증을 원자적으로 처리하여 통과 시 캐시 잔여 수량을 선점함. 조건 미달 시 RDB로 진입하지 않고 즉시 거절됨.
3. 선점 성공 시 `stock_histories` 테이블에 `DEDUCT` 이력을 Insert 한 후 성공 응답을 반환함 (**원장 기록이 API 응답보다 선행됨**).

### 4.4 재고 롤백 (API)
- 주문 취소, 결제 실패, 또는 **결제 타임아웃(Timeout)** 발생 시 호출되며, Redis 캐시 잔여 수량을 원상 복원한 후 `stock_histories` 테이블에 `ROLLBACK` 이력을 기록함.

## 5. 예외 및 안전 원칙
- 모든 시스템 예외 및 실패의 방향성은 **"재고를 보수적으로 산정하여 차감을 거절"** 하는 방식으로 설계하여 오버셀(Oversell) 발생 가능성을 원천 차단함.
- 캐시 차감 후 RDB 기록 전 크래시가 발생하는 경우 유령 차감으로 남아 보수적 거절 상태가 되며, 이는 캐시 재워밍 프로세스를 통해 정합성을 치유함.

---

## 6. 부하 테스트 및 검증 전략
- **목표 성능 검증:** 아키텍처 설계 단계부터 Redis 캐시와 Lua 스크립트 기반 구조를 바로 도입하므로, 구현 직후 k6를 활용해 대규모 동시성 요청 환경 하에서의 초당 처리량(TPS) 및 응답 속도(Latency) 한계를 측정함.
- **정합성 및 데이터 원자성 검증:** 선착순 경쟁 상황에서 오버셀이 완벽히 차단되는지, 인프라 장애 시나리오(캐시 다운 등) 상황에서 RDB 원장 기반으로 재정합성을 정상 확보하는지 집중적으로 검증함.

---

## 7. 구현 시 구체화 과제
- 내부 API의 세부 파라미터 구조(Request/Response 필드 및 데이터 타입) 확정.
- Drop 객체의 Redis 자료구조 정의, 키 네이밍 규칙 및 TTL 정책 확정.
- Lua 스크립트 내부 로직 작성 및 인자 레이아웃 설계, 예외 상황에 대한 보상 시퀀스 수립.
- 캐시 워밍 시의 동시성 제어 가드 구현 및 캐시 데이터의 RDB 반영 시점·메커니즘 구체화.
- 이력 집계 비용의 주기적 측정 및 대량 데이터 적재 시의 보강 방안 검토.
