# 담당 도메인 결정 이력 (Decision Log)

> 현재 담당하고 있는 **Product 도메인(향후 도메인 추가 시 업데이트)**의 핵심 아키텍처와 기술적 결정 사항들을 기록해 두는 공간입니다.
> 단순히 결과만 남기기보다, **왜 그렇게 판단했는지(Why)와 그 과정에서의 트레이드오프**를 담아 나중에도 의사결정의 근거를 명확히 돌아볼 수 있도록 합니다.
> 민감 정보(시크릿 키, 계정, 내부 URL 등)는 포함하지 않습니다.

## 2026-06-19

### 1. 개발 문서 구조 분리 및 관리 이원화
* **Why:** 로컬 가이드와 가공된 기술 히스토리의 성격을 분리해 전체적인 문서 관리 효율을 높이려는 목적임.
* **What:** 로컬 작업 지침(`CLAUDE.md`)과 프로젝트 개요(`docs/PROJECT.md`)의 역할을 분리하고, 도메인 아키텍처 결정 이력은 이 파일로 통합 관리하기로 함.
* **Trade-off:** 개인 설정이 포함될 수 있는 가이드 파일은 로컬에서만 제외(`git exclude`) 처리하여 형상 관리를 깔끔하게 유지하되, 프로젝트의 방향성을 담은 결정 이력은 팀의 자산이므로 히스토리가 남도록 추적하는 방향을 선택함.

### 2. 엔티티 초안 - AI, 상세 검토 - 휴먼 체크 방식 설계 
* **Why:** 초기 보일러플레이트성 스키마 작성에 드는 시간을 줄이고, 표준 구조 기반으로 설계 누락을 방지하여 생산성을 높이기 위함임.
* **What:** 초기 도메인 모델링 단계에서 표준 스키마 구조에 부합하는 초안은 AI를 활용해 빠르게 도출하고, 이후 개별 엔티티를 직접 검증하며 수정하는 방식을 채택함.
* **Trade-off:** 리소스를 아끼는 대신 AI의 제안을 맹목적으로 수용하지는 않음. 엔티티를 하나씩 직접 뜯어보며 비즈니스 요구사항에 맞게 상세 설계를 수정하고 평가하는 검증 단계를 필수로 거침으로써 시스템의 정합성을 확보함.

### 3. 카테고리 구조 설계: Enum 구조에서 RDB 테이블로 분리
* **Why:** 당장은 Enum이 편할 수 있지만, 향후 대/중/소분류 확장이나 카테고리별 유연한 수수료 정책이 도입될 수 있는 비즈니스 확장성을 고려함.
* **What:** 상품 카테고리 관리를 애플리케이션 `Enum` 대신 별도의 데이터베이스 테이블(`categories`)로 분리하여 관리하기로 함.
* **Trade-off:** 초기 개발 공수는 조금 더 들더라도, 코드 수정과 재배포 없이 데이터 변경만으로 카테고리 구조를 유연하게 바꿀 수 있는 장기적인 확장성을 챙기는 편이 유리할 수 있다고 판단함.

### 4. 테이블 네이밍 컨벤션 표준화: 엔티티 단수형 / 테이블 복수형
* **Why:** `order`, `user`, `drop` 같은 SQL 주요 예약어와의 충돌을 원천 차단하고 데이터베이스 전반의 일관된 명명 규칙을 유지하기 위함임.
* **What:** JPA 엔티티는 객체 인스턴스로 단수, 매핑되는 DB 테이블명은 복수형(예: `orders`, `users`)을 사용하기로 함.
* **Trade-off:** 공통 DB를 사용하기 때문에 팀 일관성을 위해 합의를 도출해야 하는 비용이 있음. 이유 및 근거와 대안의 장단점을 팀에 공유하고 찬/반 방식으로 전원 찬성을 유도하여 팀 컨벤션으로 승격함.

### 5. 도메인 생성 맥락 명시를 위한 의미론적 빌더(Semantic Builder) 적용
* **Why:** String이나 Instant처럼 같은 타입의 파라미터가 여러 개 존재할 때, 호출부에서 인자 순서가 뒤바뀌어 발생하는 런타임 오류를 컴파일 시점에 막기 위함임.
* **What:** 엔티티 생성 시 일반적인 정적 팩토리 메서드 대신, 행위의 의도를 담은 명명 기반의 빌더 패턴을 적용함. (예: `Product.register()...build()`, `Drop.schedule()...build()`)
* **Trade-off:** 빌더를 반환하는 구조가 관례와 다를 수 있으나, 객체 생성 코드에 비즈니스 맥락을 명시하여 가독성과 안정성을 높이는 편이 구조적으로 유리하다고 판단함.

### 6. 인덱스 도입 결정
* **Why:** PostgreSQL은 MySQL과 달리 FK 참조 컬럼을 자동으로 인덱싱하지 않음. 인덱스가 없으면 부모 행이 수정·삭제될 때마다 자식 테이블 전체를 풀 스캔하게 되므로 성능 저하를 막고 데이터 정합성을 보장하기 위한 방어 가드가 필요했음.
* **What:** 주요 조회 조건이 되는 컬럼뿐만 아니라 타 도메인을 참조하는 FK 및 타 도메인 값 참조 컬럼에 명시적으로 인덱스를 부여하기로 함.
* **Trade-off:** 과도한 인덱스로 인한 쓰기 성능 저하를 방지하기 위해 우선 단일 인덱스로 시작하고, 정렬·페이징 조건이 고정되는 시점에 복합 인덱스로 최적화하여 오버스펙 고려 필요.

### 7. 동시성 제어 전략 결정
* **Why:** 한정 수량 판매(Drop) 구조에서 선착순 재고 차감을 위한 경쟁/동시성을 위한 효율적 전략 결정이 필요하였음.
* **What:** Redis의 캐시 구조 도입으로 Drop 상태 변경/조회 관리는 캐시가 관리, 재고 변경 시 이력에 Append 방식으로 재고 경쟁 상황을 회피하고, 요청 트랜잭션을 원자적으로 처리하기 위해 Redis 내의 Lua 스크립트를 사용하기로 결정   
* **Trade-off:** 캐시 사용으로 인한 메모리를 차지할 수 있음, 그러니 비즈니스 특성상 Open 상태의 Drop 객체 개수는 한정적이므로 메모리 사용량은 미미하다 판단하여 사용 결정. (Drop 1000ea/2MB 이하 예상)

### 8. 재고 이력 멱등성을 DB 복합 유니크 제약으로 보장
* **Why:** 재시도 및 중복 요청 시 동일 주문이 재고를 중복 차감하거나 복구하는 문제를 방지하고, 애플리케이션 레벨의 체크만으로는 방어하기 어려운 동시성 레이스 컨디션을 원천 차단하기 위함임.
* **What:** 재고 이력 테이블(`stock_histories`)에 `UNIQUE(order_id, change_type)` 복합 제약을 부여하여, 동일 주문 내에서 동일 행위(DEDUCT/ROLLBACK)가 중복 INSERT되는 것을 막고 주문당 차감 1건, 롤백 1건의 멱등성을 보장하기로 함.
* **Trade-off:** 멱등성 최종 방어선을 DB 제약에 위임하여 애플리케이션 내 유효성 검증 분기를 단순화함. 다만, 제약 위반 발생 시 이미 선점된 Redis 재고를 복구하는 보상 처리 흐름이 필요하므로, 해당 예외 처리 로직을 차감 프로세스에 함께 구현하는 방향을 선택함.
