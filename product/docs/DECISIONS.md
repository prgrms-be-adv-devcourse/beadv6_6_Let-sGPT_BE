### 2. 엔티티 초안 - AI, 상세 검토 - 휴먼 체크 방식 설계

* **결정 사항 (Decision):** 초기 도메인 모델링 단계에서 표준 스키마 구조에 부합하는 초안은 AI를 활용해 빠르게 도출하고, 이후 개별 엔티티를 직접 검증하며 수정하는 방식을 채택함.

### 3. 카테고리 구조 설계: Enum 구조에서 RDB 테이블로 분리

* **결정 사항 (Decision):** 상품 카테고리 관리를 애플리케이션 `Enum` 대신 별도의 데이터베이스 테이블(`categories`)로 분리하여 관리하기로 함.

### 4. 테이블 네이밍 컨벤션 표준화: 엔티티 단수형 / 테이블 복수형
* **맥락 및 배경 (Context):** `order`, `user`, `drop` 같은 SQL 주요 예약어와의 충돌을 원천 차단하고 데이터베이스 전반의 일관된 명명 규칙을 유지하기 위함임.
* **결정 사항 (Decision):** JPA 엔티티는 객체 인스턴스로 단수, 매핑되는 DB 테이블명은 복수형(예: `orders`, `users`)을 사용하기로 함.
* **트레이드 오프 (Trade-off):** 공통 DB를 사용하기 때문에 팀 일관성을 위해 합의를 도출해야 하는 비용이 있음. 이유 및 근거와 대안의 장단점을 팀에 공유하고 찬/반 방식으로 전원 찬성을 유도하여 팀 컨벤션으로 승격함.

### 5. 도메인 생성 맥락 명시를 위한 의미론적 빌더(Semantic Builder) 적용
* **맥락 및 배경 (Context):** String이나 Instant처럼 같은 타입의 파라미터가 여러 개 존재할 때, 호출부에서 인자 순서가 뒤바뀌어 발생하는 런타임 오류를 컴파일 시점에 막기 위함임.
* **결정 사항 (Decision):** 엔티티 생성 시 일반적인 정적 팩토리 메서드 대신, 행위의 의도를 담은 명명 기반의 빌더 패턴을 적용함. (예: `Product.create()...build()`, `Drop.schedule()...build()`)
* **트레이드 오프 (Trade-off):** 빌더를 반환하는 구조가 관례와 다를 수 있으나, 객체 생성 코드에 비즈니스 맥락을 명시하여 가독성과 안정성을 높이는 편이 구조적으로 유리하다고 판단함.

### 6. 인덱스 도입 결정
* **맥락 및 배경 (Context):** PostgreSQL은 MySQL과 달리 FK 참조 컬럼을 자동으로 인덱싱하지 않음. 인덱스가 없으면 부모 행이 수정·삭제될 때마다 자식 테이블 전체를 풀 스캔하게 되므로 성능 저하를 막고 데이터 정합성을 보장하기 위한 방어 가드가 필요했음.
* **결정 사항 (Decision):** 주요 조회 조건이 되는 컬럼뿐만 아니라 타 도메인을 참조하는 FK 및 타 도메인 값 참조 컬럼에 명시적으로 인덱스를 부여하기로 함.
* **트레이드 오프 (Trade-off):** 과도한 인덱스로 인한 쓰기 성능 저하를 방지하기 위해 우선 단일 인덱스로 시작하고, 정렬·페이징 조건이 고정되는 시점에 복합 인덱스로 최적화하여 오버스펙 고려 필요.

### 7. 동시성 제어 전략 결정
* **맥락 및 배경 (Context):** 한정 수량 판매(Drop) 구조에서 선착순 재고 차감을 위한 경쟁/동시성을 위한 효율적 전략 결정이 필요하였음.
* **결정 사항 (Decision):** Redis의 캐시 구조 도입으로 Drop 상태 변경/조회 관리는 캐시가 관리, 재고 변경 시 이력에 Append 방식으로 재고 경쟁 상황을 회피하고, 요청 트랜잭션을 원자적으로 처리하기 위해 Redis 내의 Lua 스크립트를 사용하기로 결정
* **트레이드 오프 (Trade-off):** 캐시 사용으로 인한 메모리를 차지할 수 있음, 그러니 비즈니스 특성상 Open 상태의 Drop 객체 개수는 한정적이므로 메모리 사용량은 미미하다 판단하여 사용 결정. (Drop 1000ea/2MB 이하 예상)

* **트레이드 오프 (Trade-off):** 멱등성 최종 방어선을 DB 제약에 위임하여 애플리케이션 내 유효성 검증 분기를 단순화함. 다만, 제약 위반 발생 시 이미 선점된 Redis 재고를 복구하는 보상 처리 흐름이 필요하므로, 해당 예외 처리 로직을 차감 프로세스에 함께 구현하는 방향을 선택함.

* **트레이드 오프 (Trade-off):** 코드가 적은 현재는 패키지 수가 다소 늘어나는 부담이 있으나, 구조 변경 비용이 가장 싼 초기에 경계를 잡아두는 편이 장기적으로 유리하다 판단함.

### 3. 상품 모듈 전용 가이드 문서(PRODUCT.md) 분리

* **결정 사항 (Decision):** `product/docs/PRODUCT.md`를 만들어 모듈 내부 구조·컨벤션을 정리하고, 전역 규칙은 `PROJECT.md`를 참조하도록 역할을 분리(전역=`PROJECT.md` / 모듈 내부=`PRODUCT.md` / 결정 근거=`DECISIONS.md` / 재고 상세=`STOCK_GATEKEEPER.md`).
* **트레이드 오프 (Trade-off):** 관리할 문서가 하나 늘어 동기화 부담이 생기나, 전역 내용은 중복 작성 대신 `PROJECT.md` 링크로 참조해 드리프트를 줄이고, 담당 도메인 컨벤션을 한눈에 찾을 수 있는 이점을 택함.

### 4. 서비스 계층 CQRS 경량 적용(Query/Command 분리)
* **맥락 및 배경 (Context):** 비즈니스 밸리데이션 및 상태 변경을 담당하는 CUD 로직과 단순 데이터 프로젝션 위주의 R 로직이 혼재되어 발생하는 응집도 저하를 막고, 읽기 전용 트랜잭션의 `readOnly` 최적화를 명확히 적용하여 의도치 않은 상태 변경(Side Effect)을 방지하기 위함임.
* **결정 사항 (Decision):** 애플리케이션 포트와 서비스를 읽기(`XxxQueryService`)와 쓰기(`XxxCommandService`) 계층으로 명확히 분리하고, DTO 명명 규칙(`~Command` / `~Query` / `~Info`)도 분리하여 적용함.
* **트레이드 오프 (Trade-off):** 파일 및 클래스 개수가 증가하는 보일러플레이트 부담이 있으나, 이는 AI 에이전트 활용을 통해 생산성 손실을 쉽게 상쇄할 수 있음. 오히려 구조가 명확히 분리됨으로써 AI가 소스코드의 맥락을 오판하지 않고 더 정확하게 제어할 수 있는 이점을 택함. (데이터 저장소까지 분리하는 풀 CQRS는 현 단계에서 오버스펙이므로 제외)

### 5. Swagger 문서 구현 인터페이스 방식으로 결정
* **맥락 및 배경 (Context):** 팀 합의로 Swagger 도입은 확정되었으나 세부 구현 방식 결정이 필요하였으며, Swagger 어노테이션을 컨트롤러에 매핑하는 방식은 API가 많아질 시 가독성 저하가 우려됨
* **결정 사항 (Decision):** 명세 전담 인터페이스(`ProductApiSpec`)에 Swagger 관련 어노테이션을 몰아두고, 컨트롤러가 이를 `implements`하여 순수 제어 흐름만 남김. 단, 데이터 모델용 `@Schema`는 구조적 응집도를 위해 DTO record에 매핑함.
* **트레이드 오프 (Trade-off):** 메서드 시그니처를 두 곳에 유지하는 번거로움이 있으나, 컴파일 단계에서 정합성이 강제되어 명세 불일치(Drift) 위험이 없음. 상품 도메인은 향후 엔드포인트가 계속 늘어날 예정이므로 분리하는 편이 장기적으로 유리하다고 판단함.

### 6. 클라이언트 요청 ID 내부 추출 방식으로 변경
* **맥락 및 배경 (Context):** 각 클라이언트(프론트)에서 API 요청 시 구매자 id, 판매자 id 등 요청 사용자 식별 키는 JWT 인증 구조 특성상 변조에 취약함을 인지함.
* **결정 사항 (Decision):** 요청 사용자 id는 클라이언트에서 전송되는 것이 아닌 내부 JWT 인증 헤더 정보를 사용하는 것으로 변경
* **트레이드 오프 (Trade-off):** 보안 문제라 팀 컨벤션 차원에서 변경해야 하므로 각 도메인의 API 코드를 변경해야하는 트레이드 오프가 있으나, 보안 관련 부분이므로 타협할 수 없다고 생각하며, 팀에게 의견 공유 후 충돌 없이 자연스러운 합의로 변경 진행

### 1. ApiResponse 제거
* **맥락 및 배경 (Context):** ApiResponse가 사실상 data 필드만 존재하여 봉투로써의 가치가 없음을 인지(상태 코드는 HTTP status 필드와 중복), 이 경우 ResponseEntity 응답 표준 클래스를 사용하는게 유리하다고 판단함, 팀원들과 의논시 당장 공통 응답 필드 추가 예정이 없어 제거하는게 유리해보임
* **결정 사항 (Decision):** ApiResponse 제거 후 컨트롤러 응답을 ResponseEntity를 사용하도록 변경
* **트레이드 오프 (Trade-off):** API 응답 컨벤션을 변경해야하는 비용이 들어가나, API 개수와 향후 사용할 표준을 고려했을 때 지금 시점에 변경하는 것이 유리하다고 판단하였으며, 팀원의 합의를 이끌어냄.
